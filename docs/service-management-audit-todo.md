## 1. 검수 기준

서비스 관리 화면은 Docker, Compose, nginx, Harbor, certbot을 모르는 사용자를 기본 대상으로 한다. 사용자가 이해해야 하는 값은 서비스 이름, 접속 주소, 이미지 버전 정도로 제한하고, 컨테이너 이름, stack name, namespace, Compose service key, nginx upstream, published port 같은 내부 값은 Docker Infra가 생성하고 검증해야 한다.

핵심 판단 기준은 다음과 같다.

- 사용자가 선택해야 하는 값이 운영 목적에 직접 연결되는가.
- 사용자가 모르는 기술값을 입력하지 않아도 저장과 배포가 가능한가.
- 자동 생성되는 값은 중복과 충돌을 실제 배포 대상 기준으로 확인하는가.
- 실패 시 raw error가 아니라 원인 요약과 다음 행동을 보여주는가.
- 위험 작업은 영향 범위를 먼저 설명하고 되돌릴 수 있는 상태를 보존하는가.
- 화면 첫 로딩은 1초 안에 목록을 보여주고, 무거운 상세 정보는 필요할 때 나누어 불러오는가.

## 2. P0: 화면 진입 속도와 데이터 분리

### 문제

서비스 목록을 불러올 때 첫 서비스 상세까지 동기적으로 요청하고 있다. 상세 API는 Compose 원문, nginx 원문, 버전 이력, 작업 로그, 이미지 백업까지 함께 읽기 때문에 목록을 보는 데 필요 없는 IO가 초기 화면 표시를 막는다.

### 왜 중요한가

서비스 관리 화면은 사용자가 가장 자주 들어오는 핵심 화면이다. 서버와 서비스 수가 늘어나면 목록 진입이 느려지고, 사용자는 시스템 전체가 느리거나 불안정하다고 느낀다. 목록은 즉시 표시하고 상세, 고급 정보, 로그는 선택 후 비동기로 표시해야 한다.

### 완료 기준

- `/services` 진입 시 목록 API는 목록에 필요한 값만 반환한다.
- 첫 서비스 자동 선택은 하되 상세 API 응답을 기다리지 않고 목록을 먼저 렌더링한다.
- 서비스 선택 중 이전 상세 응답이 늦게 도착해도 현재 선택된 서비스 화면을 덮어쓰지 않는다.
- 상세의 Compose 원문, nginx 원문, 파일 목록, 로그는 필요 시 별도 호출 또는 지연 표시 구조로 확장 가능해야 한다.

### 작업

- [x] 목록 `load`에서 상세 `detail_service` await 제거
- [x] 서비스 상세 요청 race 방지 token 추가
- [x] 목록 API에서 생성/편집 전용 option 같은 데이터를 제거
- [x] 서비스 수정 모달을 열 때만 도메인 option 로드

## 3. P0: 서비스 초안 필수화와 기본 nginx fallback 제거

### 문제

새 서비스 생성에서 서비스 이름만 입력하면 구체적인 Compose 초안 없이도 다음 단계로 진행할 수 있고, 내부적으로 `nginx:alpine` 기본 Compose가 사용된다.

### 왜 중요한가

일반 관리자는 “nginx 컨테이너 하나”를 만들고 싶은 것이 아니라 “홈페이지”, “파일 공유”, “업무 관리” 같은 업무 목적의 서비스를 만들고 싶다. 서비스 초안 없는 기본 nginx는 개발자용 테스트 흐름에 가깝고, 사용자가 실제로 무엇을 배포하는지 이해하기 어렵다.

### 완료 기준

- 일반 생성 흐름에서는 AI 초안, Compose 직접 작성, 서버 Compose 가져오기 중 하나가 필수다.
- Compose 초안이 없으면 2단계로 넘어갈 수 없다.
- 서버 Compose 가져오기 흐름도 동일한 초안 적용 경로로 진행한다.
- API 직접 호출도 `base_content` 또는 import source가 없으면 차단한다.
- 기본 `nginx:alpine` Compose는 일반 생성 fallback으로 쓰지 않는다.

### 작업

- [x] `/services/create` 1단계 검증에 서비스 초안 필수 조건 추가
- [x] 생성 API `preflight`, `create_service`에서 Compose 초안/import source 필수 검증 추가
- [x] 생성 화면 초기 `default_content`, `default_components` 의존 제거
- [x] 서비스 초안 UI 문구를 AI/Compose 기준으로 수정

## 4. P1: Compose 초안을 사용자용 구조로 확장

### 문제

AI/직접 작성/import Compose는 다중 Compose service를 포함하지만 화면에는 `구성 1`, `구성 2` 또는 `web`, `db` 수준으로 노출될 수 있다. 어떤 구성요소가 실제 사용자가 접속하는 화면인지, 어떤 구성요소가 내부 DB/캐시인지 명확해야 한다.

### 왜 중요한가

일반 사용자는 “web 컨테이너의 80/tcp”가 아니라 “웹 화면으로 접속”을 이해한다. 서비스와 도메인 연결도 Compose 분석 결과가 공개 endpoint를 알려줘야 자동 선택할 수 있다.

### 완료 기준

- Compose 초안 분석 결과에 공개 endpoint가 명시된다.
- Compose 초안 분석 결과에 구성요소별 사용자용 이름과 설명이 있다.
- 도메인 연결 포트는 공개 endpoint를 우선 선택한다.
- 내부 DB/캐시 구성은 외부 연결 대상으로 기본 노출하지 않는다.
- 화면에는 “웹 화면”, “데이터베이스”, “캐시”처럼 의미 기반 라벨이 표시된다.

### 작업

- [x] Compose 초안 분석 결과에 `public_endpoint` 반영
- [x] Compose 초안 분석 결과에 `component_labels` 반영
- [x] `components_from_content`가 사용자용 라벨과 공개 endpoint flag를 반환
- [x] 도메인 연결 포트 radio group에서 공개 endpoint를 추천/자동 선택

## 5. P1: 비밀번호와 내부 secret 자동 생성

### 문제

서비스 초안의 DB 비밀번호가 `*_change_me` 형태로 들어 있거나 비어 있으면 사용자가 이를 직접 이해하고 바꾸기 어렵다.

### 왜 중요한가

이 서비스는 패키징 판매형 운영 장비를 목표로 한다. DB 비밀번호, root 비밀번호, 내부 app secret은 사용자가 직접 이해하거나 관리할 값이 아니라 시스템이 안전하게 생성해야 하는 값이다.

### 완료 기준

- AI/Compose 초안 결과에 자동 생성 secret 필드가 정의된다.
- 서비스 생성 시마다 secret 값이 랜덤 생성되어 Compose에 반영된다.
- 기본 화면에는 secret 값이 노출되지 않는다.
- 고급 설정에서만 필요한 경우 확인 또는 재생성할 수 있는 구조로 확장 가능해야 한다.

### 작업

- [x] AI/Compose 초안 결과에 `generated_secrets` 추가
- [x] 서비스 생성용 draft API에서 secret 값을 생성한 preview Compose 반환
- [x] 생성 화면 일반 영역에서 환경변수 raw key/value를 숨김 유지
- [x] 추후 secret 재생성/마스킹 UI를 위한 metadata 보존

## 6. P1: 원시 Docker 입력 축소

### 문제

생성/수정 화면에 이미지 이름, tag, 내부 포트, 환경변수 key/value, 볼륨 source/target이 그대로 표시된다. 일부는 고급 토글 안에 있지만 이미지와 내부 포트는 일반 흐름에 노출되어 있다.

### 왜 중요한가

사용자는 “이미지 tag”보다 “버전”, “내부 포트”보다 “연결 포트”를 이해한다. 제한을 하더라도 Compose 초안이 정한 안정적인 기본값을 쓰는 쪽이 더 안전하다.

### 완료 기준

- 일반 생성 화면에는 이미지/포트가 읽기 쉬운 요약으로만 표시된다.
- 이미지명, tag 직접 수정과 내부 포트 추가/삭제는 고급 설정 안으로 이동한다.
- 도메인 단계의 연결 포트는 Compose 초안 분석 결과가 표시한 공개 endpoint 중심으로 노출된다.
- 수정 모달도 기본 변경과 고급 변경을 명확히 분리한다.

### 작업

- [x] 생성 2단계 일반 영역에서 이미지/포트 입력 필드 제거
- [x] 생성 2단계 고급 설정 안에 이미지/포트 직접 수정 배치
- [x] 수정 모달의 이미지/포트 영역을 “버전 변경” 중심으로 문구 개선
- [x] 내부 service key, container name, namespace 문구가 기본 화면에 나오지 않도록 재점검

## 7. P1: 실제 배포 대상 기준 자동 점검

### 문제

현재 이미지 확인은 로컬 Docker와 Docker Hub 중심이고, 포트 충돌 확인은 WIZ가 실행 중인 로컬 서버 기준 socket bind로 처리된다. 실제 서비스가 다른 노드에 배치되면 점검 결과와 배포 결과가 달라질 수 있다.

### 왜 중요한가

자동화는 “화면에서 괜찮다고 했는데 실제 배포는 실패”하지 않아야 의미가 있다. 사용자가 직접 SSH로 확인할 수 없는 대상 서버의 이미지 pull 가능 여부, 포트 충돌, Docker 상태를 시스템이 대신 확인해야 한다.

### 완료 기준

- 배포 대상 노드가 결정된 뒤 해당 노드 기준으로 이미지 확인을 수행한다.
- published port는 실제 upstream 대상 노드 또는 Swarm publish 정책 기준으로 확인한다.
- 로컬 백업 시스템 registry, 로컬 이미지, Docker Hub를 순서대로 확인한다.
- 외부 registry는 경고로 통과시키더라도 배포 로그에 pull 실패 원인을 사용자용 메시지로 변환한다.
- 포트 자동 조정 결과는 Compose version metadata와 service domain metadata에 남는다.

### 작업

- [x] placement 결정 로직과 preflight를 연결
- [x] remote node docker image inspect/pull 가능성 확인 추가
- [x] remote node port 사용 여부 확인 추가
- [x] 자동 조정 결과를 version metadata에 저장
- [x] 실패 메시지를 사용자용 원인/조치 형태로 정규화

## 8. P2: 상세 화면의 고급 정보 격리

### 문제

서비스 상세 고급 정보 안에 Compose 원문, nginx 원문 편집, 컨테이너 ID, 파일 브라우저가 함께 있다. 접혀 있더라도 한 번 열면 일반 사용자가 위험한 내용을 쉽게 볼 수 있다.

### 왜 중요한가

운영자는 상태와 조치만 빠르게 확인해야 한다. 원문 편집은 장애 복구나 특수 운영 상황에서만 쓰는 기능이므로 일반 정보와 분리해야 실수 가능성이 줄어든다.

### 완료 기준

- 상세 화면 기본 영역은 접속 주소, 상태, 인증서, 백업, 최근 처리만 보여준다.
- 기술 정보는 읽기 전용으로 분리한다.
- nginx 원문 저장 같은 위험 기능은 별도 “고급 수정” 단계와 확인 모달 뒤에 둔다.
- 컨테이너 ID, raw image, operation payload는 기본 화면에서 보이지 않는다.

### 작업

- [x] 고급 정보 영역을 읽기 전용 기술 정보와 고급 수정으로 분리
- [x] nginx 원문 편집 버튼은 위험 작업 확인 모달 뒤에 표시
- [x] 컨테이너 상세 raw ID는 기본 고급 정보에서도 접거나 복사 버튼 중심으로 축소

## 9. P2: 백업과 복원 문구 정리

### 문제

이미지 백업, 현재 상태 백업, 되돌리기 버튼이 나란히 보이지만 각 동작의 영향이 충분히 드러나지 않는다. 특히 docker commit 기반 스냅샷은 pause 설정에 따라 서비스가 잠깐 멈출 수 있다.

### 왜 중요한가

백업은 안전 기능이지만 실행 방식에 따라 서비스 영향이 있다. 사용자는 “언제 잠깐 멈출 수 있는지”, “무엇을 되돌리는지”를 이해해야 한다.

### 완료 기준

- “이미지 백업”과 “현재 상태 백업”의 차이를 사용자용 문구로 설명한다.
- 현재 상태 백업 실행 전 pause 가능성과 대상 컨테이너를 확인한다.
- 복원 전 접속 주소, 이미지 버전, 구성 변경 영향을 보여준다.
- 자동 백업 정책은 기본 비활성이고, 운영 시간이 아닌 시간대를 권장한다.

### 작업

- [x] 백업 버튼 옆에 최근 백업 상태와 영향 요약 표시
- [x] snapshot 실행 확인 모달에 일시 정지 가능성 표시
- [x] 복원 영향 범위 문구를 서비스 상태 중심으로 정리

## 10. 실행 순서

1. P0 화면 진입 속도와 데이터 분리
2. P0 서비스 초안 필수화와 기본 nginx fallback 제거
3. P1 Compose 초안 메타데이터와 자동 secret
4. P1 생성 화면 원시 Docker 입력 축소
5. P1 실제 배포 대상 기준 자동 점검
6. P2 상세 고급 정보 격리
7. P2 백업/복원 문구 정리

이 순서는 사용자가 처음 만나는 흐름부터 정리하기 위한 것이다. 목록 진입과 생성 흐름이 안정되어야 이후 배포, 수정, 복원 개선도 일관되게 보인다.
